Band 能力边界

类别 Chrome 实读信息 对产品设计的含义
定位 Band 是 AI agents 的 shared operational layer，提供 persistent identity、multi-agent coordination、structured memory、unified audit trail。 产品叙事不要说“我们发明多 Agent”，要说“用 Band 把原本分散的专职 Agent 变成可审计协作流程”。
框架兼容 Agents 保留自己的 runtime、prompts、tools、LLM providers。Band 连接 LangGraph、CrewAI、Anthropic、OpenAI、Gemini、Pydantic AI 等。 Demo 可以让不同角色用不同模型或框架，展示 cross-framework collaboration。
Remote Agent Remote agents 在自己的环境运行，通过 REST 发送命令，通过 WebSocket 接收 Band 消息。 架构上应把前端、业务状态机、每个 Agent worker、Band room 分开，便于 replay 和故障降级。
平台工具 SDK 自动提供 thenvoi_send_message、thenvoi_send_event、thenvoi_add_participant、thenvoi_remove_participant、thenvoi_get_participants、thenvoi_lookup_peers、thenvoi_create_chatroom。 Demo 高光应包含主动招募专家、@mention 交接、事件记录、参与者变化，而不是只串行调用 LLM。
消息可见性 Agent 只看到明确 mention 到自己的消息。Human 可以看到全部消息。 可以把 @mention 作为可视化任务路由证据。每个 Agent 的输入边界要设计清楚。
Messages vs Events messages 用于面向参与者的文本消息，events 用于 tool calls、thoughts、errors、task progress 等信息记录。 审计链最好同时展示“对话交接”和“结构化事件”，这样更贴合评审标准。
API 视角 Human API 回答 “What’s mine?”，Agent API 回答 “Who can I work with?”。Agent key 不能访问 human endpoints。 不要让 Agent 扮演人类管理员。human approval 应做成明确节点。
测试 Band SDK 提供 FakeAgentTools，可在不连接 live platform 的情况下测试 adapter logic、tool calls、message handling。 比赛前可以先写 unit tests 和 replay fallback，Kickoff 后再接真实 Band 平台。

技术准备清单

  • Python 3.10+ 或 3.11+。Connect Any Agent 页写 3.11+，SDK Setup 页写 3.10+，实际按 3.11+ 更稳。
  • uv package manager。
  • Band account。
  • LLM provider API key，例如 OpenAI 或 Anthropic。
  • 每个 Remote Agent 在 Band 平台创建后会得到 Agent UUID 和 API key，API key 只显示一次。
  • .env 和 agent_config.yaml 必须进 .gitignore。
  • 平台 URL 示例：THENVOI_REST_URL=https://app.band.ai/，THENVOI_WS_URL=wss://app.band.ai/api/v1/socket/websocket。

Codeband 可借鉴但不宜复刻

  • Codeband 是基于 Band.ai 的 adversarial multi-model coding agents 工具。
  • 默认协作形态：Conductor、Planner、Plan Reviewer、Coders、Code Reviewers、Mergemaster、Watchdog。
  • 亮点：cross-model review、git worktree 隔离、risk-aware merging、headless/local/docker/distributed 运行。
  • 分布式模式通过 Band.ai WebSocket 和 memory 协调，代码通过 Git remote 移动。
官方把 Codeband 放在资源里，说明多 Agent 软件开发方向已有强参考实现。若参赛做该方向，必须找更细、更企业化的缺口。

从产品经理视角提炼出的设计原则

原则 为什么重要 落到产品上的形态
先选真实企业流程，不选泛 AI 工具 Business Value 和 Originality 都要求解决真实流程，而不是展示聊天能力。 选一个高摩擦流程：审批、交接、风险复核、合规调查、发布 go/no-go。
Agent 必须有角色边界 评审会看 role specialization 和 task handoff。 每个 Agent 有输入、输出、可调用工具、失败条件和升级目标。
Band 协作证据要可视化 单看最终结果很像普通 agent workflow。 展示 Band room、@mention、participant changes、events、状态机和审计时间线。
必须设计 human approval 高风险工作流不能伪装成全自动生产系统。 Human reviewer 作为签批节点，Agent 提供建议和证据，最终决策由人确认。
准备 live path 和 replay fallback Band/API/模型现场不稳定会毁掉演示。 同一 demo 场景有 live run、recorded events、replay JSON、失败回退说明。
