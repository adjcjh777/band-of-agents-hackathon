7 个 Band of Agents Hackathon 待选赛题

目标不是找“能做”的题，而是找市场上还缺、又能在一周内用 Band 展示真实多 Agent 协作的产品。下方 7 个不是最终定题，是给你手动筛选的 shortlist。

已研究方向 18 个
最终待选 7 个
统一门槛 3+ Agent through Band
共同边界 Human approval + replay fallback

筛选标准

市场稀缺
避开单点 AI、普通 chatbot、又一套 dashboard。
企业痛点
能解释成本、风险、速度、合规或收入影响。
Band 必需
协作发生在 Band 中间，而不是最终通知。
一周可演示
可做 live path，也能做 replay fallback。
可讲清楚
评委 10 分钟内能看懂角色、交接、证据链。

评分来自子代理调研与综合判断，满分 5。商业价值和 Band 适配高但市场过拥挤的方向，会被放到备选或淘汰。

最终 7 个待选项

1

Security CaseLoop

安全事件 / CVE 响应 / 合规调查的多 Agent 事实总线

4市场稀缺
5商业价值
5Band 适配
4可行性
最值得优先筛。SOAR、漏洞管理、GRC 都拥挤，但“同一个事件横跨告警、取证、CVE、合规证据的多 Agent 协作链”仍稀缺。
  • Agent 角色：SOC 分流、Forensics、Compliance/Legal、Containment/Playbook。
  • Demo 高光：注入高危 CVE，Band 中形成 timeline、证据、冲突仲裁、人工批准与最终调查包。
  • 主要竞品：Splunk SOAR、IBM QRadar SOAR、Cortex XSOAR、Microsoft Sentinel、ServiceNow SIR、Tenable、Wiz、Drata、Vanta。
  • 风险：安全与合规是高风险叙事，必须明确是 hackathon prototype，不做自动隔离或披露决定。
强推荐 Track 3 审计包 高商业痛点
2

RFP TrustRoom

RFP + 安全问卷 + 证据同源的售前协作系统

4市场稀缺
4商业价值
5Band 适配
4可行性
非常适合比赛。竞品多做 AI 草拟和内容库，但“提案、DDQ、安全问卷、证据索引在同一协作线程里可回放”仍是空位。
  • Agent 角色：需求建模、内容/语义、合规/安全、人工 SME reviewer。
  • Demo 高光：同一客户 RFP + 安全问卷，触发证据过期冲突，Band 中协商、重写、签核、生成提交包。
  • 主要竞品：Responsive、Loopio、RFP360、Qvidian、PandaDoc、OneTrust、Drata、Vanta、Whistic。
  • 风险：要避免变成“又一个问答库”，核心必须是任务状态机、证据同源和人审闭环。
强推荐 Track 1/3 售前 ROI 一周可做
3

PromiseRail

Sales 承诺到 Delivery 执行的交接控制层

4市场稀缺
4商业价值
5Band 适配
4可行性
B2B 里很真实的痛点。不要做 CS 平台，要做“销售承诺语义化包 + 交付风险签收 + CS 连续性”的控制层。
  • Agent 角色：Sales Extractor、Solution Gap、Delivery Readiness、CS Continuity。
  • Demo 高光：从 closed-won deal 抽出承诺、验收标准、依赖和风险，阻断不完整交接，生成责任时间线。
  • 主要竞品：Salesforce、HubSpot、Gainsight、ChurnZero、Planhat、Totango、Kantata、monday.com。
  • 风险：CRM/PSA/CS 字段口径复杂，MVP 要用样例数据和清晰 schema。
强推荐 Track 1 市场空位清楚 B2B SaaS
4

BugBridge

Support -> Debug -> Engineering -> Reviewer 的可修复工单闭环

4市场稀缺
5商业价值
5Band 适配
3可行性
痛点非常强，但实现要克制。不要做 triage AI，要做“复现证据包 + debug 上下文 + PR reviewer 闭环”的协作协议。
  • Agent 角色：Support Parser、Debug/RCA、Engineer Patch、Reviewer/QA。
  • Demo 高光：一条客户工单变成可复现包，触发 Sentry/GitHub mock，生成 patch 草案和 reviewer 退回/通过记录。
  • 主要竞品：Zendesk、Freshdesk、JSM、Sentry Seer、Datadog Incident AI、incident.io AI SRE、GitHub Copilot、Linear。
  • 风险：系统集成多，容易漂成“自动修 bug”。必须保留 reviewer 和 replay。
推荐 Track 1/2 集成风险 强业务价值
5

Finance CloseMesh

费用、发票、审批、月结证据链的端到端协作层

4市场稀缺
4商业价值
5Band 适配
3可行性
单模块产品很多，但“票据采集 -> 合规评分 -> 动态审批 -> 月结证据包”端到端稀缺，适合用 Band 展示多角色交接。
  • Agent 角色：票据采集、合规评分、审批编排、闭账/审计。
  • Demo 高光：金额阈值、供应商变更、票据缺失三类异常触发不同审批链，最终进入 close checklist。
  • 主要竞品：Concur、Expensify、Ramp、Bill.com、Stampli、Tipalti、FloQast、BlackLine、Trintech、金蝶、用友、钉钉智能财务。
  • 风险：财税规则和 ERP 接入复杂，比赛版要用 mock ERP 和规则兜底。
推荐 Track 1 ERP mock 审计链
6

Release GateRoom

发布 go/no-go 决策室与可 replay 的证据包

4市场稀缺
4商业价值
5Band 适配
3.5可行性
这是之前 LaunchRoom 的近亲，但不是直接定题。优势是和官方示例、评审标准高度贴合；风险是要避开 incident/CI gate 工具的同质化。
  • Agent 角色：Release Manager、Risk/QA/Security、Operations/Rollback、Human Approver。
  • Demo 高光：候选发布快照、风险争议、一票 NO-GO 阻断、回滚预案、最终决策包。
  • 主要竞品：JSM Change Management、Opsgenie、ServiceNow、Azure DevOps Gates、GitLab Approvals、Harness、LaunchDarkly、PagerDuty、incident.io、Rootly。
  • 风险：容易被评为“又一个发布面板”。必须让 Band 承载决策状态机和证据签署。
推荐 Track 1/2/3 需防同质化 评委易懂
7

Credit Approval CaseLoop

贷款 / KYC / AML / 合规审批的人机协作原型

4市场稀缺
5商业价值
5Band 适配
3可行性
高价值高风险方向。不能做真实金融建议，要定位为“多人协作审批原型 + 可审计交接 + human approval”。
  • Agent 角色：Onboarding/KYC、Risk、Policy/Pricing、Compliance Exception、Human Approver。
  • Demo 高光：正常、边缘、拒绝三类样例，Band 展示缺件、冲突规则、升级审批和 replay。
  • 主要竞品：nCino、Temenos、Mambu、Fenergo、Alloy、Trulioo、ComplyAdvantage、Pega、Unit21、Feedzai、Camunda、Appian。
  • 风险：监管敏感，文案和 demo 必须 no-overclaim，不提供真实信用/金融建议。
推荐 Track 3 合规敏感 商业价值高

评分顺序：市场稀缺 / 商业价值 / Band 适配 / 一周可行性。备选不代表差，只是相对更拥挤、风险更高，或更难证明市场空位。

我的产品经理判断

  1. 优先筛选 1-4：Security CaseLoop、RFP TrustRoom、PromiseRail、BugBridge。它们都有清楚的市场空位和强 Band 协作证据。
  2. 如果想打监管高价值：Credit Approval CaseLoop 很有冲击力，但必须把边界讲清楚，不碰真实金融建议。
  3. 如果想最贴官方与最容易讲：Release GateRoom 是好讲的题，但需要和普通 release/incident 工具拉开距离。
  4. 如果想做运营财务闭环：Finance CloseMesh 有明确企业价值，但 demo 需要 mock ERP 和月结证据包。
  5. 不要现在锁定 LaunchRoom：它仍在 7 个候选里，但不是唯一答案；我们先让你从这 7 个里筛。
